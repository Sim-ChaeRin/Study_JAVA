## 🔍 핵심 개념 정리

### 🌐 프로토콜이란?
- **프로토콜(Protocol)** 은 서로 다른 시스템 간에 통신을 하기 위한 **규약(약속된 규칙)** 이다.  
  예를 들어, 편지를 주고받을 때 발신자·수신자·주소 형식을 미리 정해두는 것과 같다.

- **HTTP(HyperText Transfer Protocol)** 은 웹에서 데이터를 주고받기 위한 대표적인 프로토콜이다.  

#### 📋 HTTP의 주요 특징
1. **텍스트 기반 프로토콜**  
   → 사람이 읽을 수 있는 형태의 요청/응답 메시지로 구성되어 있다.  
2. **비연결성(Connectionless)**  
   → 요청과 응답이 끝나면 서버와의 연결이 바로 종료된다.  
3. **무상태성(Stateless)**  
   → 서버는 클라이언트의 이전 요청 상태를 기억하지 않는다.  
     (이를 보완하기 위해 **쿠키**, **세션**을 사용한다.)  
4. **확장 가능(Extensible)**  
   → 필요에 따라 **사용자 정의 헤더(Custom Header)** 를 추가할 수 있다.

---

### 🚦 HTTP 상태 코드
| 범위 | 의미 | 설명 |
|------|------|------|
| 1xx | 정보(Informational) | HTTP/1.1부터 추가됨. 요청이 진행 중임을 알림 |
| 2xx | 성공(Success) | 요청이 정상적으로 처리됨 |
| 3xx | 리다이렉션(Redirection) | 클라이언트가 다른 위치로 이동해야 함 |
| 4xx | 클라이언트 오류(Client Error) | 잘못된 요청 (ex. 404 Not Found) |
| 5xx | 서버 오류(Server Error) | 서버 내부 문제 (ex. 500 Internal Server Error) |



### ✉️ HTTP 메시지 구조
HTTP 요청/응답 메시지는 **3부분**으로 구성된다.

1. **시작줄(Start Line)** — 요청의 종류 또는 응답 상태를 나타냄  
2. **헤더(Header)** — 부가적인 정보(길이, 타입, 쿠키 등)를 담음  
3. **빈 줄(CRLF)** — 헤더와 바디를 구분하기 위한 구분선  
4. **본문(Body)** — 실제 전송되는 데이터(HTML, JSON 등)  

> 💡 빈 줄은 헤더가 가변적이기 때문에 **헤더와 본문을 구분하기 위해 반드시 필요**하다.



### 🧭 GET과 POST 요청의 차이

| 구분 | GET | POST |
|------|-----|------|
| 목적 | 서버의 **리소스 조회** | 서버로 **데이터 전송** |
| 데이터 전송 위치 | URL(Query String)에 포함 | HTTP Body에 포함 |
| 캐싱 | 가능 | 불가능 (기본적으로 캐싱 안 함) |
| 보안성 | 낮음 (주소창에 데이터 노출) | 상대적으로 높음 |
| 대표 사용 예시 | 검색, 데이터 조회 | 회원가입, 로그인, 글 작성 |


## 📦 추가 개념 정리

### 🗂 파일의 종류
| 구분 | 설명 |
|------|------|
| **바이너리 파일 (Binary File)** | 문자와 숫자뿐 아니라 **모든 종류의 데이터(이미지, 영상, 실행파일 등)** 을 이진수 형태로 저장한 파일 |
| **텍스트 파일 (Text File)** | 사람이 읽을 수 있는 문자만 저장된 파일 (ex. `.txt`, `.html`, `.csv`) |


### 🧾 MIME (Multipurpose Internet Mail Extensions)
- **MIME** 은 HTTP 헤더 중 `Content-Type` 에 사용되어,  
  **데이터의 형식(Type)** 을 명시하는 역할을 한다.  
- 예시:
  ```http
  Content-Type: text/html
  Content-Type: application/json
  Content-Type: image/png

### 🔠 Base64 인코딩 (HTTP에서의 활용)
- Base64 는 바이너리 데이터를 텍스트 데이터로 변환하는 인코딩 방식이다.
- HTTP는 텍스트 기반 프로토콜이므로,
-  이미지나 동영상 같은 바이너리 데이터를 안전하게 전송하기 위해 Base64 인코딩을 사용한다.

### 📐 구성 원리
- 사용 문자 집합: 'A'~'Z', 'a'~'z', '0'~'9', '+', '/'
- 총 64개의 문자로 구성되어 있으며, 각 문자는 6비트(2⁶ = 64) 를 표현한다.
- 3바이트(24비트)의 바이너리 데이터를 → 4문자의 텍스트로 변환한다.

💡 예시
```html
<img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..." />

이미지를 Base64 문자열 형태로 직접 HTML에 삽입한 경우
→ 서버 파일 경로가 없어도 이미지가 표시된다.


✅ 장점
- 파일 경로가 잘못되거나 접근 불가여도 Base64 데이터 자체로 브라우저에서 즉시 표시 가능
- 외부 의존성 없는 단일 HTML 문서 구성 가능 (예: 이메일 본문 이미지 등)
  
⚠️ 단점
- 텍스트화 과정에서 파일 크기가 약 33% 증가
- 많은 이미지를 Base64로 전송할 경우 서버 저장공간 및 네트워크 전송량 증가
